01

初期セットアップ

新規マシン、または再インストール後に実行する初期化スクリプト群。 00_core/ と config/ カテゴリが対象。 原則として上から順に実行する。

スクリプト一覧
スクリプト役割
setup_ubuntu.shUbuntu 24 基本パッケージ / apt 設定。まず最初に実行。
setup_game_env.shゲームエンジン・ComfyUI 周辺の依存ライブラリを整備。
setup_rclone_meritocracia.shrclone + B2 バケット設定。B2 認証情報を事前に用意すること。
my_aliases.sh / my_functions.shシェルエイリアスと関数定義。.bashrc に source するか末尾に追記。
meritcore_start.shMeritCore CLI を起動するランチャー。config/ に配置。
meritmap.shプロジェクト全体のディレクトリマップを出力。迷子になったとき確認。
実行順序
bash setup_ubuntu.sh — 基本環境を整える。sudo が必要。
bash setup_game_env.sh — ゲーム / AI 生成環境のセットアップ。
B2 の keyID と applicationKey を手元に用意してから bash setup_rclone_meritocracia.sh を実行。
~/.bashrc に source ~/path/to/my_aliases.sh を追記し、source ~/.bashrc で反映。
bash meritcore_start.sh で CLI が起動することを確認。
⚠
WARNING
setup_rclone_meritocracia.sh は B2 バケット名を直書きしている場合がある。 実行前に中身を確認し、バケット名・リージョンが現在の設定と一致しているか確認すること。
✦
TIP
セットアップ完了後は meritmap.sh を実行してディレクトリ構造を確認しておくと、 後で「あのファイルどこ？」を防げる。
02

Backup / Restore

バックアップは 10_backup/ カテゴリと 30_project/backup_viewer_to_b2.sh が担う。 用途に応じてスクリプトを使い分ける。復元手順は後述の B2 セクションと合わせて読むこと。

スクリプト一覧
スクリプト対象優先度
backup_merito.sh Meritocracia プロジェクト全体（原稿・設定・DB） 必須
backup_viewer_to_b2.sh Viewer Hub のデプロイ済みデータを B2 に同期 必須
env_recovery_backup.sh venv / pip freeze / 環境変数スナップショット 推奨
backup_linux_settings.sh .bashrc / .gitconfig / SSH 鍵設定などのドットファイル 推奨
backup_home_mediaver2.sh ~/DOCS メディアファイル（v2 が最新版） 任意
backup_home_media.sh 同上の旧版。v2 で代替可能 legacy
backup_game_saves.sh / direct_gamesave.sh / java_gamesave.sh ゲームセーブデータ（Proton / Java ゲーム別） 任意
backup_mp4.sh 録画・動画素材 任意
cleanup_cache.sh pip / npm キャッシュ・tmp 削除 任意
毎日の最低限バックアップ
執筆セッション終了後に bash backup_merito.sh を実行。原稿・DB・JSON がローカルに保存される。
bash backup_viewer_to_b2.sh で Viewer データを B2 に同期。git push とセットで行うのが理想。
週 1 回、bash env_recovery_backup.sh で環境スナップショットを取る。pip 依存が壊れたときの保険になる。
Restore 手順（原稿・DB）
B2 バケットから rclone でファイルを取得する。
rclone copy b2:バケット名/meritocracia/ ~/DOCS/meritocracia-team-share/ --progress
Viewer の index.json が壊れている場合は bash update_viewer.sh で再生成。
RAG インデックスが古い場合は python3 rag_v2.py build を再実行。
⚠
WARNING
backup_home_media.sh（旧版）と backup_home_mediaver2.sh は同じ対象を上書きする。 両方を cron に登録すると二重実行になるので v2 のみを有効にすること。

04

Viewer 更新

Viewer Hub（GitHub Pages）の更新には複数のスクリプトが関与する。 更新対象によって使うスクリプトが異なるため、以下の分岐を把握しておく。

更新フロー全体図
→
原稿 .md 更新
↓ sync_txt_to_md.py  （.txt → .md 変換が必要な場合）
↓ update_viewer.sh  or  update_viewer_index.sh
↓ git add . && git commit -m "update" && git push
↓ GitHub Pages 自動デプロイ
↓ backup_viewer_to_b2.sh  （B2 同期）
スクリプト詳細
スクリプト用途・タイミング
update_viewer.sh Viewer 全体を再ビルドして index.json を再生成。原稿を追加・削除したときに実行。
update_viewer_index.sh index.json だけを更新する軽量版。既存ファイルの内容変更のみの場合に使う。速い。
viewer-os.sh Viewer OS（作業卓）の依存ファイルを更新するランチャー。private-schedule / ポモドーロ系の修正後に実行。
build_scripts_inventory.py scripts_inventory.json の再生成。スクリプトを追加・移動したときに実行。
sync_nation_db.sh 国家 DB の JSON を Viewer の表示用フォーマットに変換・同期。
index.json が壊れたとき
python3 -m json.tool docs/viewer/index.json でエラー箇所を特定。
手動修正が難しければ bash update_viewer.sh で全体再生成。
再生成後に git diff docs/viewer/index.json で変更内容を確認してから push。
⚠
NOTE
update_viewer.sh（全体再ビルド）と update_viewer_index.sh（index のみ）を 同時実行しないこと。競合して index.json が壊れる原因になる。

06

RAG

MeritCore の RAG（Retrieval-Augmented Generation）は プロジェクト全体の .txt / .md / .json をベクトル化して意味検索できるようにする仕組み。 60_rag/ カテゴリに 11,000 件以上あるがライブラリ依存込みの数字なので、 実際に操作するのは rag_v1.py と rag_v2.py の 2 ファイルだけ。

v1 vs v2 使い分け
rag_v1.pyrag_v2.py（現行）
キャッシュなしあり（_cached_data）
type フィルタなしあり
BOT 連携なしsearch_rag() 関数あり
推奨用途動作確認・比較用本番・BOT・全用途
インデックス作成（初回 / 更新時）
COMMAND python3 rag_v2.py build

実行すると meritocracia-team-share/ 以下の全ファイルをスキャンし、 vectors.json にベクトルデータを書き出す。 ファイル数が多い環境では数分かかる。原稿を大幅に追加した後は必ず再ビルドする。

CLI 検索
基本検索 python3 rag_v2.py "ジェイド 能力" type フィルタ付き python3 rag_v2.py "士官学校 試験制度" system

使えるフィルタ type： plot / story / character / nation / system / map / history / other

Discord BOT から使う
Discord !rag ジェイド 能力 !rag 士官学校 system

スペース区切りで第 3 引数を渡すと type フィルタになる。 引数なしならフィルタなし全検索。検索結果は上位 3 件が返ってくる。

✦
TIP — ai_prompts 除外について
ai_prompts/ ディレクトリは RAG の対象外に除外されている（load_texts() 内の skip 処理）。 AI へのプロンプトが検索結果に混入しないよう意図的に除外している。 この動作を変えたい場合は rag_v2.py の load_texts() を編集。
07

B2 三重保全

MeritCore の保全戦略は「ローカル / GitHub / B2」の三層で構成する。 どれか一つが死んでも残り二つから復元できる状態を常に維持する。

三層構成
レイヤー場所担当スクリプト更新頻度
L1 ローカル ~/DOCS/meritocracia-team-share/ backup_merito.sh 毎日
L2 GitHub meritocracia-team-share (private) git push（手動） セッション終了時
L3 B2 Backblaze B2 バケット backup_viewer_to_b2.sh / rclone sync 毎日〜週 3
B2 同期コマンド（手動実行）
Viewer → B2 bash backup_viewer_to_b2.sh ローカル全体 → B2 rclone sync ~/DOCS/meritocracia-team-share/ b2:バケット名/meritocracia/ \ --progress --exclude "*.pyc" --exclude "__pycache__/**" B2 → ローカル（復元） rclone copy b2:バケット名/meritocracia/ ~/DOCS/meritocracia-team-share/ --progress
B2 の状態確認
rclone ls b2:バケット名/meritocracia/ | head -30 rclone size b2:バケット名/meritocracia/
⚠
WARNING — rclone sync は破壊的
rclone sync は送信元にないファイルを送信先から削除する。 復元時に使う方向を間違えると B2 のデータが消える。 迷ったら削除なしの rclone copy を使うこと。
障害シナリオ別対応
障害対応
ローカルディスク障害B2 または GitHub から rclone copy で復元。env は env_recovery_backup.sh スナップショットから再構築。
GitHub リポジトリ削除ローカルから git push で再作成。B2 から原稿を補完。
B2 バケット障害ローカル + GitHub が生きていれば即復旧可能。setup_rclone_meritocracia.sh で再設定。
index.json 破損のみbash update_viewer.sh で再生成。他レイヤーへの影響なし。
vectors.json 破損のみpython3 rag_v2.py build で再ビルド。他レイヤーへの影響なし。
08

よくある事故

実際に発生した、または発生しやすいトラブルと対処法。 パニックにならず、まず状況を確認してから動くこと。

🔴 Viewer が真っ白 / カードが表示されない

原因の 90%：index.json の書式エラーか配置場所のズレ。

ブラウザのコンソールを開いて fetch エラーを確認。404 なら配置ミス、JSON parse error なら書式エラー。
python3 -m json.tool docs/viewer/index.json で構文チェック。エラー行を修正。
直しきれない場合は bash update_viewer.sh で全体再生成。
🔴 RAG 検索結果がすべて古い内容を返す

原因：原稿追加後に vectors.json を rebuild していない。または BOT がメモリに古いキャッシュを保持している。

python3 rag_v2.py build

rebuild 後は BOT を再起動する（_cached_data がメモリに残っているため）。

🔴 Discord BOT が無反応
echo $DISCORD_TOKEN でトークンの有無を確認。空なら export DISCORD_TOKEN=xxxx をセット。
プロセスが生きているか確認。
ps aux | grep bot.py
死んでいれば venv を有効化して再起動。
source venv/bin/activate && python3 bot.py
「ログイン完了」が出ない場合は Discord Developer Portal でトークンを再発行。
🔴 ComfyUI / AI 生成が起動しない

よくある原因：pip 依存破損 / CUDA バージョンずれ / モデルファイルのパスずれ。

python3 generate.py でエラー内容を確認。ModuleNotFoundError なら pip 再インストール。
pip install -r requirements.txt --break-system-packages
モデルパスのエラーなら ai_model_select.sh でパスを再設定。
それでも直らない場合は env_recovery_backup.sh のスナップショットから環境を復元。
🟡 rclone sync で B2 のファイルを誤削除した

Backblaze B2 はデフォルトでバージョニングが有効。削除されたファイルも一定期間は「隠しファイル」として残る。

B2 Web コンソール → バケット → 「Show hidden files」を有効にして削除されたファイルを探す。
対象ファイルを選択して「Restore」。または rclone --b2-versions フラグで旧バージョンを取得。
ℹ
INFO
バージョニングの保持期間は Lifecycle Rules で決まる。 設定は B2 コンソール → バケット → Lifecycle Rules で確認。
🟡 schedule.json に不正な JSON が入った

Discord BOT の !schedule で特殊文字・改行が混入した場合。

python3 -m json.tool schedule.json でエラー行を特定。
エラー箇所の前後のエントリを手動で確認・削除。
再度 python3 -m json.tool でエラーがなくなったことを確認してから BOT を再起動。
🟡 merit-daily-v4.sh が途中で止まる

スクリプト内部で呼んでいる外部コマンドが存在しない・パスが違う場合。

bash -x merit-daily-v4.sh 2>&1 | head -50 でデバッグ出力を確認。どの行で止まるかを特定。
存在しないパスが原因なら meritmap.sh で現在の構造を確認してスクリプト内のパスを修正。
